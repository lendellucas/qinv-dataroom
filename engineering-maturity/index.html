1. Engineering summary

2. Environments & software development lifecycle

QINV maintains three strictly isolated environments, separated at the network level. Changes progress through the lifecycle in one direction, from experimentation to validation to production, so that production stability is never coupled to active development.

Development

Sandbox for initial feature work and experimentation. Isolated from production data and traffic.

Homologation

Pre-production mirror used for QA and integration testing. Validates changes against a production-like surface before release.

Production

Live environment serving end users with the highest stability guarantees and the tightest access controls.

3. CI/CD pipeline

Releases are automated and auditable. Source control governs every change, container images are built and versioned in a private registry, and deployment is an atomic image swap that supports hotfixes without full downtime.

Source control
Build
Image registry
Versioned artifact
Deploy
Atomic swap

Pipeline stages

  1. Source control: enterprise Git with role-based permission boundaries and CI/CD hooks.
  2. Artifact build: Docker images built and pushed to a private container registry.
  3. Deployment: versioned image tag swap, enabling atomic releases and rapid rollback.

Release governance

  • Changes are reviewed before promotion across environments.
  • Production deployment is a controlled, reversible operation.
  • Image versioning makes every release traceable to a known artifact.
  • Hotfixes ship as atomic swaps without a full-service teardown.

4. Reliability & disaster recovery

QINV is engineered for recoverability. Data is backed up daily, infrastructure state is replicated to a secondary region, and recovery follows a "cattle, not pets" model in which a standby environment is rapidly rehydrated from known-good restore points.

Automated backups

Daily automated backups of all database instances ensure data persistence and integrity.

Geo-redundancy

Infrastructure state is replicated to a secondary region through restore-point collections.

Failover strategy

Rapid hydration of a standby environment from restore points, minimizing recovery time.

Design principle: infrastructure is treated as reproducible and disposable. Recovery rebuilds from versioned artifacts and restore points rather than relying on manually maintained, irreplaceable servers.

6. Security posture

QINV operates a Zero Trust model with a default deny-all policy: traffic is blocked unless explicitly allowed. Only the client-facing web layer is publicly reachable; databases, the optimization engine, and orchestration interfaces are not exposed to the public internet.

Network isolation

Default deny-all. The public perimeter is limited to the encrypted web surface; all other entry points are blocked.

Private resources

Databases, algorithms, and orchestration interfaces are invisible to the public internet and reachable only through governed private access.

Role-based access control

Access is scoped by role. Operations roles hold broader privileges; engineering roles are constrained to non-production environments.

Privileged access

Administrative access to private resources requires governed, encrypted connectivity rather than open exposure.

7. Security & data governance

QINV enforces a strict separation between commodity infrastructure and proprietary intelligence. Third-party providers are used only for standardized infrastructure, while all core decision-making logic remains proprietary and self-hosted. Personal data and financial data are held in separate silos that cannot be cross-referenced by unauthorized personnel.

Air-gapped intelligence

No external vendor or AI provider has access to a user's optimized portfolio or investment strategy. The financial intelligence layer is fully isolated from third-party APIs. LLMs receive only anonymized statistics to render into natural-language summaries, never raw market data or personal information.

Two-silo data model

Identity data and financial data are stored separately and linked only by an anonymized identifier. A reviewer auditing the algorithm sees portfolio data tied to an opaque user ID, never the underlying personal identity.

GDPR & LGPD aligned

The architecture supports automated compliance with data-deletion ("right to be forgotten") requests.

Cascade deletion

Invalidating an identity triggers automated removal of all associated financial records, leaving no residuals.

No training leakage

No user data is sent to external AI models for training, eliminating third-party "model memory" risk.

8. Key architectural decisions

QINV's engineering choices favor operational simplicity and auditability without sacrificing rigor. Two decisions illustrate the engineering judgment behind the platform.

Compute: containers on VMs, not Kubernetes

"Given team size and performance needs, the operational overhead of Kubernetes outweighed the benefits."

  • Simpler operations: roughly 90% of containerization benefits at significantly lower complexity.
  • Deterministic performance: avoids "noisy neighbor" effects of shared clusters, ensuring consistent algorithm execution.

Task queue: transactional SQL queue, not a broker

"For portfolio optimization, job visibility and auditability matter more than raw throughput."

  • Job visibility: full transactional insight into every job's status inside the persistence layer.
  • Debug & recovery: failed jobs can be inspected and retried with standard SQL.
  • Transactional integrity: keeping job state with data simplifies ACID guarantees.

These are deliberate trade-offs, not defaults. They reflect an engineering culture that optimizes for reliability, auditability, and lean-team velocity over fashionable complexity.
